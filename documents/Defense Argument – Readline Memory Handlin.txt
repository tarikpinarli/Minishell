Defense Argument – Readline Memory Handling
During testing with Valgrind, our shell reports some memory as still reachable at exit, specifically from the readline() library. This is expected and acceptable behavior, as confirmed in the Minishell subject PDF:

“The readline() function may cause memory leaks, but you are not required to fix them.”

We ensured that all memory allocated in our own code is properly freed, and Valgrind confirms this by showing:

0 bytes definitely lost

0 bytes indirectly or possibly lost

The only memory still reachable originates from the internals of readline() and associated libraries like libhistory or termcap, which manage line buffers and history. These cannot and should not be manually freed by the user, as doing so would violate the abstraction provided by the library.

Therefore, the reported “still reachable” memory blocks are safe, intentional, and allowed under project rules. No leaks exist in our implementation.
